Chapter 01 · Foundations

HTML is the Skeleton

Before any JavaScript runs, you need a place for everything to live. HTML provides the structure — the boxes, labels, buttons, and empty containers that JavaScript will later fill in.

Think of an HTML document as a set of nested containers. Every element lives inside another element, forming a tree. Our BMI app needs: a way to type in weight, a way to type in height, a button to trigger the calculation, and an area to display the result.

The nesting structure of our app

Diagram — App HTML Tree
<div class="bmi-app"> — the outer wrapper
<div class="unit-toggle">
Two buttons: Metric / Imperial. Toggles the unit system.
<div class="weight-input">
Label + number input for weight. Has an id="weight" handle.
<div class="height-input">
Label + number input for height. Has an id="height" handle.
<button id="calc-btn">
The trigger. Clicking this runs the calculation.
<div id="result-panel"> display:none initially
Hidden until JS adds the "show" class. Contains the BMI number, category, gauge, and recipes.

Why the result panel is hidden at first

There's nothing to show before the user has entered values. Rather than removing and re-adding the panel element, we simply hide it with CSS and reveal it with JavaScript once the calculation is done. The element always exists in the HTML — JS just controls whether it's visible.

🔑
The ID attribute is your handle Every element you need to interact with gets a unique id. Think of it like a name tag — JavaScript calls getElementById('weight') to grab that specific element and nothing else. Class names style groups of things; IDs identify individual things.
⚛️
React Connection In React you'd write this same structure as JSX — it looks almost identical to HTML. The key difference is that React generates the HTML for you based on state, so you don't manually write id attributes to find things. You hold references in variables called refs, or you just let state drive what's visible.

The DOM — Your Map of the Page

The DOM (Document Object Model) is the browser's live representation of your HTML. JavaScript doesn't read the HTML file — it talks to the DOM, which is a tree of objects sitting in memory.

Real World
A building's floor plan
Shows every room, where doors are, what's inside each room
≈
In the Browser
The DOM
Shows every element, how they're nested, what's inside each one

Querying — asking "where is this element?"

Before you can do anything to an element, you need a reference to it — a variable that points to it. The process of finding an element in the DOM is called querying.

Diagram — getElementById in action
HTML (in your file)
<input id="weight" type="number">
<input id="height" type="number">
<button id="calc-btn">Go</button>
→
JS Variables (your handles)
weightInput → the weight <input>
heightInput → the height <input>
calcBtn → the <button>

The ID in HTML becomes a handle in JavaScript — same name, different world.

💡
Do this once at the top Querying the DOM is like looking up someone's phone number. You don't look it up every time you want to call them — you save it once and reuse it. Store all your DOM references at the very top of your script, before any other logic.
⚛️
React Connection React mostly removes the need for manual DOM queries. Instead of asking "where is this input?", you create a controlled input — the input's value is stored in state, and React keeps the DOM in sync automatically. You only use refs (React's version of a DOM handle) in rare cases, like focusing an input programmatically.
Chapter 03 · Foundations

Variables — Named Boxes for Data

A variable is simply a named container. You put a value in, give it a name, and refer to that name whenever you need the value. JavaScript has three keywords for creating variables — each with different rules.

Real World
A labelled storage box
"Winter clothes" — you know what's inside without opening it
≈
In JavaScript
A variable
const userName = "Alice" — the name holds the value

The three keywords compared

Diagram — const vs let vs var
Keyword Can change? Scope When to use in this project
const ❌ Never reassigned Block DOM references — weightInput, calcBtn. These always point to the same element.
let ✅ Can change Block The isMetric flag — flips between true/false when the user toggles units.
var ✅ Can change Function (leaky) Don't use it. Legacy syntax with confusing behaviour. Always prefer const or let.

What our script stores in variables

Diagram — Our app's variable inventory
  • weightInput const Points to the weight <input> element
  • heightInput const Points to the height <input> element
  • calcBtn const Points to the Calculate button
  • resultPanel const Points to the hidden result container
  • bmiNumber const Points to the element that will show "23.4"
  • bmiCategory const Points to the element that will show "Normal Weight"
  • recipeList const Points to the container where recipe cards are injected
  • isMetric let Starts as true. Flips to false when Imperial is selected.
⚠️
Why all DOM references use const const means the variable can't be reassigned — you can't point it at a different element. But the element itself can still change (its text, colour, classes). So const weightInput is correct: the variable always points to the same input, even though the input's value changes as the user types.
⚛️
React Connection In React, const is everywhere. Every component function re-runs on each render, so all variables inside it are declared fresh each time — const is almost always the right default. State values (managed by useState) are the equivalent of let variables that persist across renders.

Our app's two styles of function

Diagram — Traditional vs Arrow function

The BMI Formula — Reading Inputs & Doing Maths

This is where the calculation happens. Three things to understand: reading a value from an input, converting it from a string to a number, and applying the formula.

The "string trap" — why input values need converting

Every value you read from an HTML input arrives as a string of text — even if the user typed a number. The browser doesn't know you want maths. parseFloat() converts "70" (text) into 70 (a number you can do arithmetic with).

The 703 constant bridges the unit gap between the metric and imperial systems.

🧮
Rounding the result Raw BMI calculations produce long decimals like 22.857142.... We use .toFixed(1) to trim to one decimal place, giving us 22.9. But toFixed returns a string again — so we wrap it in parseFloat() to get back to a number.
⚛️
React Connection In React, you'd read weight and height from state variables (not from DOM inputs directly). The formula itself is pure JavaScript maths — it doesn't change at all. The difference is just where the values come from: state instead of .value.

Filtering — pulling out only the matching recipes

.filter() is an array method that takes every item, runs a test on it, and returns a new array containing only the items that passed. Think of it as a colander — water (non-matching) drains away, pasta (matching) stays.

Template literals — building HTML strings

When we inject recipe cards, we need to build HTML markup from our data. Template literals let you embed variable values directly inside a string using ${ } placeholders — no clumsy string joining required.

Diagram — Template literal anatomy
// A recipe card built from an object called "recipe"
`<div class="recipe-card">
  <h4>${recipe.name}</h4>
  <p>${recipe.calories} kcal</p>
</div>`
${ } placeholders JavaScript evaluates the expression inside and drops the result into the string. Works with variables, object properties, function calls — anything.
Backtick delimiters Template literals use backticks (`) not quotes. This lets them span multiple lines, making HTML structure readable.
⚛️
React Connection React replaces innerHTML injection entirely. Instead of building an HTML string, you return JSX — which looks like HTML but is actually JavaScript. React then handles inserting it into the DOM efficiently. The ${ } pattern becomes { } in JSX — same concept, cleaner syntax.

The SVG Gauge — Drawing with Code

The semicircular BMI gauge is built with SVG — a drawing format that lives right inside your HTML. Every visual element is a programmable object, making it perfect for data visualisation.

What SVG actually is

Real World
Architectural drawings
Precise instructions for shapes, lines, curves. Scale to any size without blurring.
≈
In the Browser
SVG elements
Precise XML shapes in your HTML. Scale infinitely. Fully interactive with JS and CSS.

The three layers of the gauge

Diagram — Gauge anatomy (exploded view)
  • 1
    Track arc — The grey background semicircle. A static SVG <path> element. It defines the shape of the gauge and never changes.
  • 2
    Fill arc — The coloured arc that grows as BMI increases. Uses the stroke-dashoffset trick to reveal itself gradually. JavaScript animates this.
  • 3
    Needle — A line that rotates around the gauge centre. JavaScript sets its rotation angle based on the BMI value using CSS transform: rotate().

The stroke-dashoffset trick — how the fill arc "draws itself"

Diagram — strokeDashoffset from 414 (hidden) → 0 (fully drawn)
offset = 414
fully hidden
BMI ~15
offset = 290
30% visible
BMI ~22
offset = 165
60% visible
BMI ~30
offset = 0
fully visible
BMI 40+

414 is the arc's total length in pixels (π × radius). The CSS transition property makes the change animate smoothly rather than jumping.

Mapping BMI to an angle — the needle's maths

Diagram — BMI value → needle angle (−90° to +90°)
Points left
Points straight up
Points right

The full sweep is 180°. We map the BMI's position within the 15–40 range to a proportional position within that −90° to +90° range.

Placing zone ticks with trigonometry

To draw the tick marks at BMI 18.5, 25, and 30 at the correct position on the arc, we use polar coordinates. We know each tick's angle, and we use the sine and cosine of that angle to find the corresponding x and y screen coordinates.

Diagram — Polar to cartesian coordinate conversion

Given a centre point, a radius, and an angle, the coordinates of any point on that circle are:

x = cx + r × cos(angle)
y = cy − r × sin(angle)

The minus sign on y is because SVG's y-axis points downward, which is the opposite of standard maths convention.

📐
You don't need to memorise the maths The formula x = cx + r×cos(θ) and y = cy − r×sin(θ) is something you look up when you need it. What matters is understanding why it works: you're describing a point's location by its distance and direction from a centre, then converting that to a flat x/y grid.
⚛️
React Connection SVG works identically in React — you write <path>, <line>, <circle> as JSX. Instead of manually setting style.strokeDashoffset via DOM queries, you'd store the BMI in state and compute the offset as a derived value on every render: const offset = 414 - bmiToFraction(bmi) * 414 — then pass it directly as a prop.
